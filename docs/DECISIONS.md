## 2026-05-31: Realize Historical Backfill P&L At The First Executable Suggested Close

Historical research/backfill tracked positions must use the first executable historical suggested-close point for realized P&L, not a later lifecycle placeholder close. The close must be backed by trusted exact-contract OPRA/NBBO spread evidence for the selected legs. Stop, target, profit-harvest/giveback, and time-exit checks may close a row during historical repair only when the pessimistic executable spread price can be computed from exact bid/ask rows.

Rows with no executable exit quote stay visibly missing rather than being filled with midpoint, last-trade, stale, daily/EOD, or synthetic values. The unresolved rows are data-quality work items: either import trusted exact exit-leg quotes for the specific OCC contract/date, or keep them out of realized production proof.

## 2026-05-31: Use Profit-First 90% Stops For Live Tracked Positions

The tracked-position review engine should optimize for the strategy's expected profit, not for conservative account-risk optics. For the current paper/live-shadow use case, the configured `stop_loss_pct=90` is intentional: rows down more than `50%` can remain `Hold` if the tested strategy's profit profile expects that much option drawdown before a mechanical stop.

Durable decision: live tracked-position review now uses a `90%` effective stop cap/default instead of `50%`. A configured stop at or below `90%` is honored; stops wider than `90%` are capped to `90%` for live review. Stop and target triggers still require an executable bid/ask-derived exit quote, so display-only last-price marks remain informational and must not auto-close positions. If an executable review produces a zero exit price, paper tracking should close at `0.0` rather than leaving a total-loss option open indefinitely.

Saved position-review metrics must include `review_policy_version` and `max_live_review_stop_loss_pct` so future audits can distinguish current 90% profit-first reviews from stale historical review text generated under older stop caps.

The positions UI must not call the state-changing review endpoint during passive page load or background polling. Opening the app and reading positions should be read-only; users must explicitly refresh/review before the UI POSTs to `/api/positions/review` or `/api/suggested-trades/review`.
